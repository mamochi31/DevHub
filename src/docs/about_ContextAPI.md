# Context API について

Reactの **Context API** は、アプリケーション全体で**「グローバルな状態（Global State）」**を管理し、コンポーネント間でデータを簡単に共有するための組み込み機能です。

## なぜ必要なのか？（Prop Drilling の解決）

Reactでは通常、親コンポーネントから子コンポーネントへ「Props」を使ってデータを受け渡します。
しかし、アプリが大きくなると、深く階層化されたコンポーネントにデータを渡すためだけに、中間のコンポーネントが不要なPropsをバケツリレーのように渡し続ける現象が発生します。
これを **Prop Drilling（プロップ・ドリリング）** と呼びます。

Context API を使うと、このバケツリレーをスキップして、データの提供元から直接データを受け取ることができます。

## 3つの基本要素と役割

### ① `createContext` (コンテキストの作成)
データを共有するための「箱（空間）」を作ります。
（※ TypeScript環境では、型定義と初期値 `undefined` をセットにするのが一般的です）

### ② `Provider` (データの提供)
作った箱の中に実際のデータ（Stateや関数）を入れ、それを子孫コンポーネントに配る役割を持ちます。
`<Provider value={...}>` で囲んだ内側にあるコンポーネントなら、どれだけ階層が深くてもデータにアクセスできます。

### ③ `useContext` (データの受け取り)
Providerが提供している箱の中から、データを取り出して使うためのフックです。
（※ 実務では、これを取り出すための専用のカスタムフック `useTheme()` などをProviderと同じファイルで作成することが推奨されます）

---

## 💡 補足：Context API の注意点

### 向いているデータ
アプリ全体で共有する必要があり、かつ**頻繁には更新されないデータ**に最適です。
- UIテーマ（ダークモード / ライトモード）
- ログイン中のユーザー情報
- アプリの言語設定（i18n）

### 向いていないデータ
**頻繁に値が書き換わるデータ**には不向きです。

**【理由：再レンダリングの影響】**
Contextの `value` が更新されると、そのContextを `useContext` で参照している**すべてのコンポーネントが強制的に再レンダリングされる**という仕様があります。頻繁に変わる値を入れると、アプリ全体が重くなる原因になります。
（※ 更新頻度が高い場合は、Zustand や Redux といった外部の状態管理ライブラリを検討します）
